What‘s up with this ?

blaxxun Contact 3D (blaxxunCC3D or CC3D in short) is an effort to implement the VRML 97 Standard (ISO/IEC 14772-1:1997). The specification can be found at http://www.web3d.org/technicalinfo/specifications/vrml97/index.htm

Beside VRML 97, VRML 1.0 and some Open Inventor nodes are supported.

Overview

blaxxun Contact 3D currently is only available for the WIN32 platform using either OpenGL or Direct 3D as low level rendering API. There is a core library providing rendering, utility, media support and VRML scene graph services. On this core the sources for two different front-end’s are provided:

blaxxuncc3d.ocx An MFC Based Active X Control as a plugin for Active X Applications (e.g. Microsoft Internet Explorer)

GLView.exe A standalone MFC Based application including some editing capabilities and a SceneGraph Tree Viewer.

A Netscape Navigator plugin is provided which is realized as a small wrapper for blaxxuncc3d.ocx.

The EAI is internally realized using COM (blaxxunVRML.idl)

cc3dglut A standalone simplistic hello world application for Linux using the Mesa 3.1 & GLUT libary.

History

blaxxunCC3D originally started as the GLView project, a 3D OpenGL Viewer for Windows NT. Over time support for VRML 1.0 and VRML 2.0 support was added. The original front end is a standalone application with some editing capabilities, other front ends are an MFC-Based Active X control.

 

How to build it

Requirements

Microsoft Visual C / Developer Studio Version 5.0 or 6.0 (with Service Packs recommended)

Microsoft Direct X 7.0 SDK (header & lib files are provided for a safer build environment in /client/cc3d/dxsdk5)

Microsoft Direct X Media SDK (header & lib files are provided for a safer build environment in /client/cc3d/dxsdk5/dxmedia)

For building the „Katmai" Build version with Intel Pentium III support in gkatmai.cpp Intel 1.0 SDK including the Intel C/C++ compiler. (headers & libs can be copied to /client/cc3d/intel)

 

 

Installation of source

The zip files are containing absolute paths and need to be copied to a common directory, the CC3D branch starts then in /client/cc3d.

Compilation

Currently there are only build versions for Windows 95/98/NT, all output files will be generated in the hardcoded C:\cc3d branch.

Building the blaxxunCC3D OCX Control

For Microsoft Visual C 5.0

Open the project workspace /client/cc3d/blaxxuncc3d/blaxxuncc3d.dsw

For Microsoft Visual C 6.0

The VC 5.0 workspace will be automatically converted.

There are build variants for OpenGL/ Direct X, debug / release , with/without Katmai (PIII)

After choosing the build configuration blaxxunCC3D.ocx is build and automatically registered. Opening a wrl file in Microsoft Internet Explorer will automatically launch the built OCX control. Debugging is possible by using "Default Web Browser" in "Executable for debug session" in the MSVC Project - Debug settings.

Building the GLView standalone application

For Visual C 5.0

Open the project workspace /client/cc3d/src/glview.dsw In addition the jpeg and zlib static Build variants need to be build manually.

The executable GLView.exe or GLViewd3d.exe will be built.

Building the Linux cc3d VRML library

Please follow the latest instructions in the toplevel README files. GNU Maketools, GNU compiler, Mesa, Glut are required.

Dependencies

There are subprojects for building the supporting jpeg and zlib libraries.

The jpeg support library (/client/cc3d/src/jpeghgnt/jpeg.dsp) contains in addition to the libjpeg source (Independent JPEG Group's software Version 6a 7-Feb-96) some image processing functions and the libpng (libpng 1.0 beta 3) files.

zlib is located in (/client/cc3d/src/core/zlib and of version 1.0.4, Jul 24th, 1996).

The source & support lib for the Real Player G2 interface is not included at this time because of license restrictions of the Real G2 SDK.

Opening the subprojects with Visual C 6.0 will convert the provided VC 5.0 DSP files to VC 6.0 the first time.

Extraction on Linux

On Linux the source can be extracted using
unzip -aa cc3d_src.zip

in order to convert any CR-LF terminated files to Unix LF convention.

Possible things to do

Here are some ideas to extend features or fix flaws in CC3D

Porting to other Platforms.

The core library (sub directories of src) need to be ported.

A new front end need to be written (standalone program, Netscape plugin) using the UI and the UI-3D Window connection provided by that platform.

Starting using the OpenGL Version is recommended.

Some major platform dependencies :

www.cpp - HTTP downloading (may use Netscape Plugin stream_as_file mechanism)

Media support - Media support can be incrementally added turn off options in gconfig.h first

Sound - A new sound device driver need to be written (derive from GSoundDevice, GSoundEmitter)

gl/copengl.cpp - connecting an OpenGL context to a window is platform specific

gmodel.cpp - probably this module is now to complex and has some WIN32 / dependencies

Its is planned to modularize gmodel.cpp and the Navigation module.

gurlcache.cpp some WIN32 file function - need to be ported

gpanel.cpp retrieving the resource bitmap from a resource's need to be ported

See the libCC3D/stdafx.h for emulation of WIN32/MFC (CriticalSection, CString, CTime) etc dependencies

mak.bat & libCC3D.mak are adapted to compile the core using cygwin tools.

Adding a new node

Look at the Contact 4.2 class overview document.

Look for a suitable node pattern, write the module for the node and add the node to the node table in GvNode.cpp

 

Adding stereo rendering

RenderState – add state functions for enabling/disabling stereo , in GView::DrawScene – draw the scene twice for the left-right eye views

Adding Java in the script node support

Find functions how to interface to the Java VM : possible options are Kaffee, using MS-Java, Netscape liveconnect.

Write a classloader for an URL, write a custom GvConnection derived class (e.g. GvScriptFunctionConnection for vrmlscript) propagating evenIn’s to the Java Script.

There is some COM support for Script, this was intended for implementing Java in the Script Node for Microsoft Java. All related code is in GvScript.cpp.

Extending the JavaScript interpreter

The embedded interpreter is based on the SGI vrmlscript interpreter. A new object model and some JavaScript features and bugfixes had been added.

To make the interpreter more JavaScript / ecmascript compatible one can implement additional language features like the Data class, check the implementation of the Array Class. The switch, for (var i= and other statements are missing. Another problem is that in some other implementations of JavaScript Authors can be lazy about putting ; statement terminators in the code.

Adding Componentization

One idea is to allow native extension modules implementing new nodes. The way could be, to create a new variant of GvExternProto allowing loading of dynamic libraries. The library exports a node factory.

In order to allow the native module to do useful stuff, the native module need to access the Scene Graph or Browser internal datastructures (Rendering context, traversal context etc.)

One idea is simply making CC3D core available as dynamic linked library, the component can directly access the scenegraph using the existing scene graph API.

A second idea is to define a set of abstract interfaces for Browser, Node, Field TraversalContext etc and that the native extension is only allowed to use this services.

In addition GvScript has some initial code for allowing COM based objects as Scripts. Here the required interfacing is minimal (uses the existing COM-EAI). This would allow "Computation Engines" as native plugins, if the script node is wrapped in a PROTO this is also an option of having custom node components.

Further componentization would break down the browser in more blocks like different image / video processing modules, pluggable renderes.

This has not done yet for the following reasons:

  • Linking everything into one binary reduces the overall size, smart linkers eliminate unneeded functions.
  • It has less distribution hassles and has fewer dependencies on version of installed DLL's etc.
  • Compilers can optimize the code better, inlining, less virtual function call overhead, global optimizations.
  • Support for Direct X 3 - Execute buffers is difficult to abstract in a generic rendering driver interface.

 

Possible things to fix

  • ProximitySensor and VisibilitySensor are triggered during rendering – better they should be triggered before rendering. This has not been implemented because of speed concerns.
  • Group nodes are not automatically computing bounding boxes and are only doing rendering culling if the author provided explicit bboxSize values. Implementing this needs a propagation of scene graph changes in order to force re-update of internal bounding boxes.
  • the Direct Media support is limited (no streaming, no spatialized sound) see GvTexture.cpp

 

Questions

Christopher K. St. John" cstjohn@quik.com asked :

> Note that I completely understand that Blaxxun is under no obligation
>to release its source code under any sort of open license. But the Web3D
>consortium must understand that any non-Open Source license is unlikely
>to interest the Open Source community.

Linda Hahner answered :

This is absolutely accurate. The Open Source community is only interested in
Open Source technology. The Web3D Consortium is interested in advancing
VRML97 our ISO Standard. blaxxun's willingness to place their source code
into Community Source enables the Consortium to move ahead with our schedule
and goals and to keep our X3D development on track. Just as you may not be
interested in working for free, I do not think that we can ask companies to
simply give away technology they have heavily invested in or have spent a
lot of money to buy. The blaxxun Community Source agreement...allows
developers like me to use this browser, alter it, etc and to deliver it to
my clients for free. (If I charge I need to have a different licensing
arrangement with blaxxun.) For those of us who have felt at the mercy of
browser companies for so long...this is a very welcome development. For
those who are working with Open Source technology and have strong feelings
about this...it may fall short of your expectations. I hope that we can
agree to disagree. The Consortium has a responsibility to the Internet
community to not drop the ball on our development efforts. Even though I
personally would lke to provide the community with Open Source
technology...I don't happen to own any;-)

When our member companies come forward and offer very generously to share
their technology, we need to make certain that the spirit with which this
agreement was entered into is fully understood. Right now the Consortium has
both Open and Community Source agreements with the following companies--Sun
Microsystems, NVIDIA, and blaxxun. All of these agreements are enabling us
to move toward our goal of creating a ubiqutious Web3D standard.
